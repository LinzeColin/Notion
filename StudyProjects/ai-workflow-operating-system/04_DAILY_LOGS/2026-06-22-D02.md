## D02 Core Question

What context should Codex load so it does not run off-course, forget baseline, or waste tokens?

## Core Learning

A Context Contract is not "give the agent more background". It is a rule for what the agent must read, must ignore by default, should trust first, and must prove it actually used.

The minimum unit is not the whole chat history. The minimum unit is a context item with:

| Field | Meaning |
|---|---|
| source | file, Notion page, GitHub artifact, memory, user message, CI/log, etc. |
| authority | whether this source can override other sources |
| freshness | current, stale, historical, or unknown |
| purpose | why this context is needed for this run |
| evidence | path, URL, line, commit, log, or artifact handle |
| exclusion rule | when not to load it |

## Context Authority Ladder

For complex Codex work, use this default order:

1. Latest explicit user instruction in the current turn.
2. Current repo/project `AGENTS.md` and hard safety rules.
3. Live source files, tests, CI, logs, artifacts, and current GitHub state.
4. Current `HANDOFF.md`, `state.json`, Task Pack, acceptance files.
5. Notion page content when it is the linked notebook or source of user notes.
6. Memory summaries and old conversation summaries as routing context only.
7. Old chat history, old ZIPs, old reports, and browser planning notes as weak context unless revalidated.

If these conflict, Codex should stop and report the conflict instead of averaging them.

## Context Budget Rules

Default startup budget for a complex task:

| Slot | Required? | Example |
|---|---:|---|
| Current user goal | yes | the latest user message |
| Project rules | yes | `AGENTS.md` |
| Current state | yes | `HANDOFF.md` or `state.json` |
| Acceptance target | yes when present | tests, CI, task pack acceptance |
| Evidence layer | yes when claiming completion | diff, logs, artifacts, PR/CI |
| Old memory | conditional | only for routing or prior decisions |
| Full repo scan | no by default | only when the target requires discovery |
| Full chat replay | no by default | too costly and drift-prone |

## Context Contract v1

```text
Context Contract v1

Goal:
- Use the minimum context needed to complete this run without losing baseline.

Must load:
1. Latest user request.
2. Project/root AGENTS.md when available.
3. Current HANDOFF.md or state file when available.
4. Task Pack / acceptance criteria / tests when the task is implementation or delivery.
5. Latest evidence source before claiming completion.

Must not load by default:
1. Entire chat history.
2. Whole repository scans.
3. Old ZIPs, old reports, or old planning notes.
4. Unrelated Study Projects or unrelated repos.
5. Large logs unless a failure requires them.

Must output before action:
- context_used table: source, why loaded, authority, freshness, evidence path, risk.

Stop if:
- user goal, AGENTS.md, state/HANDOFF, and live files conflict;
- required acceptance criteria cannot be found;
- context is stale but would change the decision.
```

## Practical Use

The user should not need to provide this template manually. A good Codex run should reconstruct this contract automatically from project files. The user's low-cost control phrase is:

```text
停，先给 context_used。
```

This phrase means: before continuing, show what context you loaded, why, what authority it has, and what you intentionally excluded.

## Verification Thinking

A context contract is not verified by the agent saying "I checked". It is verified by visible evidence:

- exact file paths or URLs;
- commit SHA or artifact path;
- test/log/CI handle;
- short `context_used` table;
- explicit excluded context.
