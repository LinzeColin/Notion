## Core Explanation Notes

### What is it?

RAG means Retrieval-Augmented Generation. Plainly: before the model answers, the system retrieves relevant evidence from a controlled knowledge base, then gives that evidence to the model as context.

For this project, the key sentence is:

> RAG is not “让模型记住资料”; RAG is “先找证据，再生成答案”。

### Core theory

A practical RAG system usually has this chain:

1. Source documents or data descriptions are collected.
2. Content is parsed and split into chunks.
3. Each chunk is converted into embeddings.
4. Embeddings are stored in a vector database or retrieval index.
5. A user query is converted into a query embedding.
6. Retrieval finds likely relevant chunks.
7. Optional reranking filters and orders the evidence.
8. The LLM answers using the retrieved context.
9. Citations and evals check whether the answer is supported.

The important judgment is not whether this architecture sounds modern. The important judgment is whether the workflow needs evidence retrieval and whether retrieval can measurably reduce time, risk, cost, or error.

### How should I use it?

Use RAG when the answer must be grounded in specific documents, internal knowledge, SOPs, manuals, logs, cases, or policies.

Do not use RAG as a magical substitute for clean structured data, deterministic rules, real-time monitoring, quantitative calculation, workflow execution, permissions and audit controls, simulation and optimization, or domain expert validation.

### How it improves ability, productivity, thinking, or efficiency

A good RAG system can reduce repeated search time, lower onboarding cost, speed up troubleshooting, preserve expert knowledge, and make AI outputs auditable.

A bad RAG system only creates confident text with weak evidence.

### ROI-max action

For Industrial AI SaaS, the first ROI-max action is to map one high-frequency workflow where workers waste time searching or explaining information, then ask whether RAG can reduce that time with traceable evidence.

Scoring: 1 = weak, 5 = strong. Strong RAG candidate usually has high evidence need, available sources, high repeated search/troubleshooting cost, and high auditability need.

|---|---:|---:|---:|---:|---:|---:|---|
| Operator asks why an alarm happened and wants similar historical incidents | 5 | 4 | 4 | 4 | 5 | 5 | Strong RAG: retrieve SOP + alarm history + incident reports. |
| Maintenance engineer searches equipment manual + past work orders | 5 | 5 | 4 | 4 | 4 | 5 | Strong RAG: high repeated search cost and clear sources. |

| Approving a maintenance request through repeated steps | 3 | 3 | 3 | 4 | 4 | 3 | Workflow automation first; RAG can support policy lookup. |
| Optimizing fuel usage/yield under constraints | 3 | 3 | 5 | 2 | 4 | 4 | Simulation/optimization first; RAG retrieves prior cases and constraints. |
| Generating post-incident report from logs, SOPs, and operator notes | 5 | 4 | 5 | 3 | 5 | 5 | Strong RAG + structured data candidate; separate evidence from conclusions. |

### Scorecard decision rule

Use RAG when this sentence is true:

> The user needs a natural-language answer grounded in specific internal evidence, and the business value comes from faster search, better explanation, training, troubleshooting, or auditability.

Avoid RAG-first when this sentence is true:

> The task is mainly real-time measurement, exact calculation, deterministic workflow execution, control optimization, or structured database query.

## Required Output 2 - Decision Matrix

| Tool / approach | Best for | Bad for | Industrial SaaS example | Main failure mode |
|---|---|---|---|---|
| Model memory | General explanation, broad concepts, brainstorming | Company-specific facts, current state, auditability | Explain what “rotary kiln” generally means | Confident but unverifiable answer. |
| RAG | Evidence-grounded answers from documents/logs/manuals/cases | Real-time exact data, control actions, optimization | Find similar overheating incidents and explain likely causes with sources | Wrong retrieval, stale docs, unsupported answer, fake citation. |
| Fine-tuning | Changing model behavior/style or repeated task format | Injecting frequently changing factual knowledge | Make assistant use plant-specific terminology and response format | Model sounds aligned but still lacks fresh evidence. |
| SQL / rules | Exact structured queries and deterministic decisions | Fuzzy document understanding | Show last 24h kiln temperature above threshold | Schema mismatch, missing data, brittle rules. |
| Workflow automation | Repeated process steps, approvals, routing, notifications | Open-ended reasoning without context | Auto-create maintenance ticket when alarm condition persists | Automates wrong process or lacks exception handling. |
| Agent execution | Multi-step task with tools, planning, and action | Unsafe high-impact actions without constraints | Draft investigation plan, query logs, prepare report, ask for approval before action | Tool misuse, permission leak, runaway action, no hard boundary. |
| Simulation / optimization | Testing strategies, control policies, what-if analysis | Explaining unstructured documents by itself | Simulate fuel/yield tradeoff under constraints | Model assumptions not matching plant reality. |

### Decision rule for the user

Before building, ask in this order:

1. Is the main problem evidence search/explanation? If yes, consider RAG.
2. Is the main problem exact current data? Use DB/time-series/query.
3. Is the main problem repeated steps? Use workflow automation.
4. Is the main problem autonomous tool use? Use agent with hard permissions and fail-closed controls.
5. Is the main problem prediction/control/optimization? Use modeling/simulation/evals.
6. Is the main problem model tone/format? Consider fine-tuning only after prompt/templates fail.
